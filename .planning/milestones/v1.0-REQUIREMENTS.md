# Requirements: VitaClaw Iteration 2

**Defined:** 2026-03-26
**Core Value:** 比用户记得更全、追得更连续、整理得更系统——在需要就医或决策时，能快速调出完整上下文。

## v1 Requirements

### Engineering Foundation

- [x] **ENG-01**: Project has pyproject.toml with core dependencies and optional extras per skill group
- [x] **ENG-02**: All shared modules can be imported via standard `import` without sys.path hacks
- [x] **ENG-03**: CI pipeline runs pytest on push and reports test results
- [x] **ENG-04**: ruff linter configured and passing on all shared modules
- [x] **ENG-05**: Data migration tool can detect format version and migrate existing health data forward

### User Onboarding

- [x] **ONBD-01**: New user can complete guided health profile setup through conversational flow (demographics, conditions, medications, goals)
- [x] **ONBD-02**: Onboarding populates USER.md, IDENTITY.md, and _health-profile.md with structured data
- [x] **ONBD-03**: User can re-run onboarding to update their profile at any time

### OCR Pipeline

- [x] **OCR-01**: User can provide a photo/PDF of a medical document and get structured data extracted (体检报告, 检验单, 门诊病历, 处方单)
- [x] **OCR-02**: Extracted fields are displayed to user for confirmation before storage
- [x] **OCR-03**: Confirmed data is auto-stored into corresponding HealthDataStore records and items files
- [x] **OCR-04**: OCR supports Chinese medical document table layouts via PP-StructureV2 table extraction
- [x] **OCR-05**: Original document image/PDF is archived in patient records for reference

### Device Import

- [x] **IMPT-01**: User can import Google Fit data from Google Takeout export (CSV + TCX files)
- [x] **IMPT-02**: User can import Huawei Health data from data export (CSV/JSON/GPX in ZIP)
- [x] **IMPT-03**: User can import Xiaomi/Mi Fitness data from account data export (CSV in ZIP)
- [x] **IMPT-04**: All importers use shared base class with fuzzy deduplication to prevent duplicate records across devices
- [x] **IMPT-05**: Imported data maps to existing HealthDataStore JSONL format (heart rate, steps, sleep, activities)

### Visit Summary

- [x] **VISIT-01**: User can generate a doctor-ready visit summary with one command
- [x] **VISIT-02**: Summary includes recent vitals with trends, current medications, recent lab results, key concerns
- [x] **VISIT-03**: Summary output available as Markdown (editable), HTML (phone display), and optional PDF (print)
- [x] **VISIT-04**: Summary embeds trend charts for key metrics (blood pressure, blood glucose, weight)

### Visualization

- [x] **VIZ-01**: User can generate trend charts for blood pressure (dual-line systolic/diastolic), blood glucose, weight, and sleep
- [x] **VIZ-02**: Charts include clinically meaningful reference range bands (e.g., normal BP range)
- [x] **VIZ-03**: Charts correctly render CJK characters (Chinese labels, axis text)
- [x] **VIZ-04**: Charts can be generated for configurable time ranges (7d, 30d, 90d, 1y)

### Correlation Analysis

- [x] **CORR-01**: User can request cross-metric correlation analysis (e.g., sleep vs blood glucose, medication timing vs BP)
- [x] **CORR-02**: Correlation results include statistical confidence level and sample size
- [x] **CORR-03**: Results are presented as natural language insights (e.g., "When you sleep <6h, next-day glucose averages 15% higher")

### Annual Report

- [x] **ARPT-01**: User can generate a year-in-review health report
- [x] **ARPT-02**: Report includes metric trajectories, medication adherence stats, health events timeline, doctor visits, improvement areas
- [x] **ARPT-03**: Report output as styled HTML with embedded charts

### Family Management

- [x] **FAM-01**: User can create and switch between family member profiles (e.g., self, parent, child)
- [x] **FAM-02**: Each person's health data is isolated in per-person data directories
- [x] **FAM-03**: Existing single-user data auto-migrates to "self" profile without data loss
- [x] **FAM-04**: All skills (tracking, OCR, import, visualization) work in per-person context
- [x] **FAM-05**: person_id field is threaded through HealthDataStore, HealthMemoryWriter, and CrossSkillReader

## v2 Requirements

### Enhanced OCR

- **OCR-V2-01**: AI-powered field extraction maps lab items to LOINC codes via ConceptResolver
- **OCR-V2-02**: OCR auto-suggests which skill should track each extracted metric

### Proactive Alerts

- **ALRT-01**: Heartbeat auto-detects statistically significant cross-metric patterns and surfaces them
- **ALRT-02**: Alert frequency is tuned to avoid fatigue (only actionable patterns)

### Conversational Profiling

- **ONBD-V2-01**: Onboarding uses AI-driven conversational flow instead of structured prompts
- **ONBD-V2-02**: Follow-up questions adapt based on user responses

## Out of Scope

| Feature | Reason |
|---------|--------|
| Real-time API sync with wearable platforms | Violates local-first architecture; requires OAuth server |
| Built-in camera UI | VitaClaw is a skill library, not a standalone app |
| FHIR/HL7 hospital EHR integration | Massive compliance burden, beyond personal tool scope |
| Diagnostic AI / treatment recommendations | Medical liability; SOUL.md explicitly prohibits |
| Cloud sync / multi-device access | Local-first architecture; users can use filesystem-level sync |
| Interactive web dashboards (D3.js/Plotly) | Adds JS stack complexity; matplotlib sufficient |
| Wearable real-time streaming | Requires Bluetooth stack and background services |
| Gamification (badges/streaks) | Trivializes health management for target persona |
| Quick input shortcuts (Widget/Telegram bot) | Not needed this iteration |
| Push channel expansion (WeChat/email) | Existing channels sufficient |
| Heartbeat cross-platform deployment | macOS launchd sufficient currently |
