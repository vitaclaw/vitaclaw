# Phase 4: Clinical Output - Research

**Researched:** 2026-03-26
**Domain:** Doctor-ready visit summaries and annual health reports with multi-format output
**Confidence:** HIGH

## Summary

Phase 4 delivers two clinical output features: (1) an enhanced visit summary that replaces the existing `generate_visit_briefing.py` with a new `HealthVisitSummary` class producing Markdown/HTML/PDF output with embedded charts, and (2) a new `HealthAnnualReport` class generating year-in-review HTML reports with metric trajectories, medication adherence, events timeline, and correlation insights. Both outputs must be person_id-aware and degrade gracefully with sparse data.

The existing codebase provides strong foundations: `HealthChartEngine` generates PNG charts for BP/glucose/weight/sleep, `CorrelationEngine` discovers cross-metric correlations, `HealthDataStore` provides JSONL data access with person_id, `CrossSkillReader` aggregates across skills, and `HealthMemoryWriter` manages file output. The main gap is the existing `HealthVisitWorkflow` lacks person_id threading and chart embedding. The new modules need to compose these existing building blocks into polished, self-contained HTML output.

**Primary recommendation:** Build two new modules (`health_visit_summary.py` and `health_annual_report.py`) that compose existing engines rather than rewriting them. Use Jinja2 (already installed, v3.1.2) for HTML templating. Embed charts as base64 PNG. Generate self-contained single-file HTML with inline CSS.

### Pattern 1: Compositor Pattern
**What:** New modules compose existing engines rather than reimplementing data access. `HealthVisitSummary` calls `HealthChartEngine`, `HealthDataStore`, `HealthMemoryWriter`, and `HealthHeartbeat` to gather data, then renders output.
**When to use:** Always for these new modules -- never re-query JSONL directly when an engine exists.
**Example:**
```python
class HealthVisitSummary:
    def __init__(self, data_dir=None, person_id=None, workspace_root=None, memory_dir=None):
        self.person_id = person_id
        self.chart_engine = HealthChartEngine(data_dir=data_dir, person_id=person_id)
        self.writer = HealthMemoryWriter(
            workspace_root=workspace_root, memory_root=memory_dir, person_id=person_id,
        )
        # Data stores for direct queries where no engine exists
        self._stores: dict[str, HealthDataStore] = {}

    def generate(self, days=30, format="markdown", write=True) -> dict:
        data = self._collect_data(days)
        charts = self._generate_charts(days)
        if format == "html":
            return self._render_html(data, charts, write)
        elif format == "pdf":
            return self._render_pdf(data, charts, write)
        return self._render_markdown(data, charts, write)
```

### Pattern 2: Base64 Chart Embedding
**What:** Generate chart PNG via HealthChartEngine, read bytes, base64-encode, embed as `<img src="data:image/png;base64,...">` in HTML.
**When to use:** All HTML output that includes charts.
**Example:**
```python
import base64

def _embed_chart(self, chart_path: str) -> str:
    """Convert chart PNG file to base64 data URI for HTML embedding."""
    if not chart_path:
        return ""
    path = Path(chart_path)
    if not path.exists():
        return ""
    img_bytes = path.read_bytes()
    b64 = base64.b64encode(img_bytes).decode("ascii")
    return f"data:image/png;base64,{b64}"
```

### Pattern 3: Graceful Data Degradation
**What:** Every section in visit summary and annual report handles missing data without crashing. Show "insufficient data" message instead of empty or error.
**When to use:** Always -- users may have only partial data.
**Example:**
```python
def _vitals_section(self, days: int) -> dict:
    bp_records = self._query_bp(days)
    section = {"has_data": bool(bp_records)}
    if not bp_records:
        section["message"] = "近期无血压记录"
        section["chart"] = ""
    else:
        section["records"] = bp_records
        section["chart"] = self.chart_engine.generate_blood_pressure_chart(days)
    return section
```

### Pattern 4: Self-Contained HTML
**What:** HTML output uses inline CSS (no external stylesheets), base64 images (no external files), and no JavaScript. Single `.html` file that works offline.
**When to use:** All HTML output from both visit summary and annual report.

### Anti-Patterns to Avoid
- **Direct JSONL parsing in new modules:** Always use `HealthDataStore.query()` or `CrossSkillReader.read()` -- they handle person_id filtering, time ranges, and format normalization.
- **External CSS/JS in HTML output:** Violates the self-contained requirement. Everything inline.
- **Crashing on missing data:** User may have zero records for a metric. Always handle gracefully.
- **Reimplementing chart generation:** Use `HealthChartEngine` -- it already handles CJK fonts, reference bands, and correct formatting.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Chart generation | Custom matplotlib code | `HealthChartEngine.generate_*()` | Already handles CJK fonts, reference bands, file paths |
| Correlation analysis | Custom stats code | `CorrelationEngine.discover_correlations()` | Handles Pearson/Spearman auto-selection, p-values, NL output |
| Data aggregation | Direct JSONL file reads | `HealthDataStore.query()` / `CrossSkillReader.read()` | person_id filtering, time ranges, format normalization |
| HTML templating | String concatenation | Jinja2 templates | Cleaner conditionals, loops, filters for complex HTML |
| PDF generation | Custom PDF library | weasyprint (optional) | Renders HTML faithfully; graceful fallback to browser print |
| Base64 encoding | Manual byte manipulation | `base64.b64encode()` | stdlib, one-liner |
| Risk/concern flags | Custom risk detection | `HealthHeartbeat.run(write_report=False)` | Existing issue detection with priority levels |

**Key insight:** This phase is primarily a composition and rendering task. All data access and analysis engines exist from Phases 1-3. The new code should focus on data collection, section assembly, and multi-format rendering.

## Common Pitfalls

### Pitfall 1: HealthVisitWorkflow person_id Gap
**What goes wrong:** The existing `HealthVisitWorkflow` uses `patient_id` (for archive bridge) but has NO `person_id` support for data store queries. Simply wrapping the existing class will produce wrong data for family members.
**Why it happens:** Phase 1 added person_id to data stores but didn't retrofit the visit workflow.
**How to avoid:** The new `HealthVisitSummary` class must thread `person_id` through all data queries independently. Do NOT try to retrofit `HealthVisitWorkflow` -- build fresh composition.
**Warning signs:** Visit summary for "mom" shows self's data.

### Pitfall 2: HealthChartEngine Import Context
**What goes wrong:** `HealthChartEngine` uses `from health_data_store import HealthDataStore` (non-package import). When imported from a script that has `skills/_shared` on sys.path this works, but relative imports fail.
**Why it happens:** Phase 3 decision to use direct HealthDataStore access instead of CrossSkillReader to avoid relative import issues.
**How to avoid:** The new modules in `skills/_shared/` should use relative imports (`from .health_chart_engine import HealthChartEngine`) but be aware that HealthChartEngine internally does a non-relative import. Ensure `skills/_shared` is on sys.path in CLI scripts.
**Warning signs:** `ModuleNotFoundError: No module named 'health_data_store'` at runtime.

### Pitfall 3: weasyprint Installation Complexity
**What goes wrong:** weasyprint requires system-level dependencies (cairo, pango, GDK-PixBuf) that are non-trivial to install. Users who don't have these get confusing errors.
**Why it happens:** weasyprint is a Python wrapper around C libraries.
**How to avoid:** PDF generation must be wrapped in try/except ImportError. Return a clear message: "Install weasyprint for PDF support, or open the HTML file and print to PDF from your browser."
**Warning signs:** `OSError: cannot load library` on weasyprint import.

### Pitfall 4: Large Base64 Images in HTML
**What goes wrong:** Multiple high-DPI chart PNGs embedded as base64 can produce very large HTML files (10MB+), making the file slow to open on mobile.
**Why it happens:** HealthChartEngine uses DPI=150 with figsize=(12,6). Each chart PNG can be 200-400KB, which becomes ~300-500KB as base64.
**How to avoid:** Use DPI=100 for embedded charts (lower than standalone). Limit chart count. Consider using a smaller figsize for embedded charts. The annual report with 4-5 charts at DPI=100 should stay under 3MB total.
**Warning signs:** HTML file exceeds 5MB.

### Pitfall 5: Daily Log Event Extraction
**What goes wrong:** "Health events timeline" for annual report requires parsing free-text daily markdown logs to find notable events (hospitalizations, new diagnoses, medication changes).
**Why it happens:** Daily logs are unstructured markdown written by the AI during conversations.
**How to avoid:** Use keyword matching approach: scan for keywords like "住院", "诊断", "手术", "新增用药", "停药", "急诊", "检查结果" in daily log lines. Also check structured sections in items/ files (appointments.md, medications.md) for date-tagged events. Don't try to do NLP -- simple keyword + section matching is sufficient.
**Warning signs:** Empty events timeline despite months of daily logs.

### Pitfall 6: Template File Location
**What goes wrong:** Jinja2 templates stored relative to module file work in development but break in installed package context.
**Why it happens:** `__file__` resolution differs between development and pip-installed packages.
**How to avoid:** Store templates as Python string constants within the module file itself, or use `importlib.resources` for package-aware resource loading. Given the project's non-package import pattern, inline string templates are safer.
**Warning signs:** `TemplateNotFoundError` when running from a different working directory.

### Pitfall 7: Missing Health Profile Data
**What goes wrong:** Visit summary section 1 (Patient header: name, age, conditions) reads from `_health-profile.md`. If onboarding was never completed, all fields say "pending".
**Why it happens:** User may not have run onboarding.
**How to avoid:** Check for "pending" values and substitute with "未填写" or omit the field. Never show raw "pending" in clinical output.
**Warning signs:** Visit summary shows "Name: pending, Date of birth: pending".

```python
def _collect_data(self, days: int = 30) -> dict:
    """Gather all data needed for visit summary sections."""
    start = (datetime.now() - timedelta(days=days)).isoformat(timespec="seconds")

    # Section 1: Patient header from health profile
    profile = self._parse_health_profile()

    # Section 2: Recent vitals from data stores
    reader = CrossSkillReader(data_dir=self.data_dir)
    vitals = {
        "bp": reader.read("blood-pressure", start=start, person_id=self.person_id),
        "glucose": reader.read("blood-sugar", start=start, person_id=self.person_id),
        "weight": reader.read("weight", start=start, person_id=self.person_id),
    }

    # Section 3: Current medications from items file
    medications = self._read_medications()

    # Section 4: Recent lab results from data store
    labs = self._read_lab_results(start)

    # Section 5: Risk flags from heartbeat
    heartbeat = HealthHeartbeat(data_dir=self.data_dir, person_id=self.person_id)
    issues = heartbeat.run(write_report=False).get("issues", [])

    return {
        "profile": profile,
        "vitals": vitals,
        "medications": medications,
        "labs": labs,
        "issues": issues,
    }
```

```python
import base64
from pathlib import Path

def _charts_as_base64(self, days: int) -> dict[str, str]:
    """Generate charts and return as base64 data URIs."""
    charts = {}
    for metric, method_name in [
        ("bp", "generate_blood_pressure_chart"),
        ("glucose", "generate_blood_glucose_chart"),
        ("weight", "generate_weight_chart"),
    ]:
        path = getattr(self.chart_engine, method_name)(days)
        if path:
            img_bytes = Path(path).read_bytes()
            b64 = base64.b64encode(img_bytes).decode("ascii")
            charts[metric] = f"data:image/png;base64,{b64}"
        else:
            charts[metric] = ""
    return charts
```

### Medication Adherence Calculation
```python
def _medication_adherence(self, year: int, person_id: str | None = None) -> dict:
    """Calculate medication adherence as percentage of days with dose records.

    Returns {medication_name: {total_days, tracked_days, adherence_pct, monthly_breakdown}}.
    """
    start = f"{year}-01-01T00:00:00"
    end = f"{year}-12-31T23:59:59"
    reader = CrossSkillReader(data_dir=self.data_dir)
    doses = reader.read("medication", start=start, end=end, person_id=person_id)

    # Group by medication name, then by month
    by_med: dict[str, dict[str, set]] = {}
    for record in doses:
        med_name = record.get("data", {}).get("medication", "unknown")
        date = record.get("timestamp", "")[:10]
        month = date[:7]
        by_med.setdefault(med_name, {}).setdefault(month, set()).add(date)

    # Calculate adherence per medication
    result = {}
    for med_name, months in by_med.items():
        total_days = 0
        tracked_days = 0
        monthly = {}
        for month_str, dates in sorted(months.items()):
            days_in_month = _days_in_month(month_str)
            total_days += days_in_month
            tracked_days += len(dates)
            monthly[month_str] = round(len(dates) / days_in_month * 100)
        result[med_name] = {
            "total_days": total_days,
            "tracked_days": tracked_days,
            "adherence_pct": round(tracked_days / max(total_days, 1) * 100),
            "monthly_breakdown": monthly,
        }
    return result
```

### Health Events Extraction from Daily Logs
```python
_EVENT_KEYWORDS = [
    ("住院", "hospitalization"),
    ("出院", "discharge"),
    ("手术", "surgery"),
    ("诊断", "diagnosis"),
    ("新增用药", "new_medication"),
    ("停药", "medication_stop"),
    ("急诊", "emergency"),
    ("检查结果", "test_result"),
    ("复查", "followup"),
    ("血压偏高", "bp_alert"),
    ("血糖偏高", "glucose_alert"),
]

def _extract_events_from_daily_logs(self, year: int) -> list[dict]:
    """Scan daily markdown logs for notable health events using keyword matching."""
    events = []
    daily_dir = self.writer.daily_dir
    if not daily_dir.exists():
        return events

    for log_file in sorted(daily_dir.glob(f"{year}-*.md")):
        date = log_file.stem  # e.g., "2025-06-15"
        content = log_file.read_text(encoding="utf-8")
        for keyword, event_type in _EVENT_KEYWORDS:
            if keyword in content:
                # Extract the line containing the keyword for context
                for line in content.splitlines():
                    if keyword in line:
                        events.append({
                            "date": date,
                            "type": event_type,
                            "keyword": keyword,
                            "context": line.strip()[:120],
                        })
                        break
    return events
```

### PDF Generation with Graceful Fallback
```python
def _render_pdf(self, html_content: str, output_path: str) -> dict:
    """Generate PDF from HTML. Falls back gracefully if weasyprint unavailable."""
    try:
        from weasyprint import HTML
        HTML(string=html_content).write_pdf(output_path)
        return {"path": output_path, "format": "pdf", "success": True}
    except ImportError:
        # Save HTML and instruct user
        html_path = output_path.replace(".pdf", ".html")
        Path(html_path).write_text(html_content, encoding="utf-8")
        return {
            "path": html_path,
            "format": "html",
            "success": False,
            "message": (
                "weasyprint 未安装，已生成 HTML 文件。"
                "请在浏览器中打开并使用"打印为 PDF"功能。"
                "安装命令: pip install weasyprint"
            ),
        }
```

## Open Questions

1. **Jinja2 template storage location**
   - What we know: Jinja2 is installed (v3.1.2). Project uses `sys.path` imports, not pip packages.
   - What's unclear: Whether to use inline Python string templates or external `.html` files.
   - Recommendation: Use inline string constants within the module files. This avoids template-not-found issues across different execution contexts and keeps everything self-contained. The templates are moderately complex but manageable as Python strings with Jinja2 syntax.

2. **Annual report "year" parameter with partial data**
   - What we know: User may request 2025 report but only have data from October-December.
   - What's unclear: Should we show 12 months with "no data" for 9 of them, or compress to only active months?
   - Recommendation: Show all 12 months in trajectory charts (empty periods visible as gaps). In text summaries, state "tracking started in October 2025 (3 months of data)" prominently. This is more honest than compressing.

3. **CrossSkillReader vs direct HealthDataStore for visit summary**
   - What we know: Phase 3 correlation engine used direct HealthDataStore to avoid relative import issues. CrossSkillReader uses relative imports internally.
   - What's unclear: Whether the new modules should follow Phase 3's pattern or use CrossSkillReader.
   - Recommendation: Use CrossSkillReader since both new modules live in `skills/_shared/` and can use relative imports. This is more maintainable and handles concept resolution automatically.
