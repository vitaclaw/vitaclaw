# Phase 4: Clinical Output - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Doctor-ready visit summaries (one command → Markdown/HTML/PDF with embedded charts) and comprehensive annual health reports (year-in-review with metric trajectories, medication adherence, events timeline). Both outputs deliver the core value: complete context when it matters most.

</domain>

<decisions>
## Implementation Decisions

### Visit Summary
- **D-01:** Enhanced visit summary replaces existing `generate_visit_briefing.py`. New module `skills/_shared/health_visit_summary.py` with class `HealthVisitSummary` that aggregates data from multiple sources and generates multi-format output.
- **D-02:** Summary sections in order: (1) Patient header (name, age, conditions), (2) Recent vitals with trend charts (BP, glucose, weight — last 30d), (3) Current medications + dosages + adherence notes, (4) Recent lab results (from OCR-ingested data), (5) Key concerns and risk flags (from heartbeat/memory), (6) Questions for the doctor (user-editable section).
- **D-03:** Output formats: Markdown (primary, editable), HTML (mobile-friendly with embedded base64 chart PNGs, clean medical styling), PDF (via HTML print — `weasyprint` if available, otherwise instruct user to print from browser). No new heavy dependency required.
- **D-04:** Charts embedded in HTML as base64 PNG images generated by `HealthChartEngine` from Phase 3. Markdown version references chart file paths.
- **D-05:** Visit summary is person_id aware. "Generate visit summary for mom" produces a summary scoped to mom's data.
- **D-06:** Exposed as `skills/health-visit-summary/SKILL.md` + CLI `scripts/generate_visit_summary.py`.

### Annual Report
- **D-07:** Annual report module `skills/_shared/health_annual_report.py` with class `HealthAnnualReport`. Generates a comprehensive year-in-review report.
- **D-08:** Report sections: (1) Year overview (key stats: total records, doctor visits, tracking days), (2) Metric trajectories (BP, glucose, weight, sleep — 12-month trend charts), (3) Medication adherence timeline (months of consistent tracking vs gaps), (4) Health events timeline (notable events from daily logs — hospitalizations, new diagnoses, medication changes), (5) Improvement areas (metrics that improved vs worsened, with specific numbers), (6) Correlation insights (top 3 significant correlations from correlation engine), (7) Goals review (from _health-profile.md goals vs actual progress).
- **D-09:** Output as styled HTML with embedded charts. Single self-contained HTML file (inline CSS, base64 images). No external dependencies at view time.
- **D-10:** Styling: clean, modern, medical-professional aesthetic. Light background, clear section headers, data tables with alternating row colors. Mobile-responsive. Print-friendly CSS.
- **D-11:** Annual report is person_id aware. Each family member can have their own annual report.
- **D-12:** Exposed as `skills/health-annual-report/SKILL.md` + CLI `scripts/generate_annual_report.py`.

### Claude's Discretion
- HTML template design details (exact colors, fonts, spacing)
- How to extract "health events" from daily logs (keyword matching vs structured parsing)
- Whether to use Jinja2 templates or Python string formatting for HTML generation
- Medication adherence calculation method (consecutive days with records vs percentage)
- How to handle missing data in annual report (e.g., metrics only tracked for 3 months)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `HealthVisitWorkflow` in `health_visit_workflow.py`: Has pre-visit briefing generation. Can be enhanced or replaced.
- `HealthChartEngine.generate_*()`: Directly callable for embedding charts in summaries/reports.
- `CorrelationEngine.discover_correlations()`: Provides top correlation insights for annual report.
- `CrossSkillReader.read_all()`: Aggregates data across skills for comprehensive reporting.

### Established Patterns
- CLI scripts: argparse with --format markdown|json, --person-id, --days
- SKILL.md with YAML frontmatter for OpenClaw discovery
- Base64 chart embedding: `HealthChartEngine` saves PNG, read bytes and base64-encode for HTML

### Integration Points
- Visit summary reads from HealthDataStore (vitals, labs) + HealthMemoryWriter (medications, concerns) + HealthChartEngine (charts)
- Annual report reads from all the above + correlation engine + daily logs + health profile
- Both write output to `memory/health/files/` for archival

</code_context>

<specifics>
## Specific Ideas

- Visit summary is the "killer feature" — it's what users show their doctor. Must be clean, professional, and complete.
- Annual report is the "engagement feature" — it's what makes users feel that continuous tracking was worth it.
- Both must work with sparse data (user may only have 3 months of tracking) — degrade gracefully.
- HTML output must be a single self-contained file — no external CSS/JS/image dependencies.

</specifics>
