## 0. 北极星目标

VitaClaw 的终局不应该只是：

- 技能数量很多
- 医疗方向覆盖很广
- 能做几个健康分析 demo

VitaClaw 的终局应该是：

1. **开箱即用**
   用户安装后，不需要自己设计人格、记忆结构、目录结构、长期跟踪逻辑，就能直接拥有一个可用的健康分身。

2. **长期连续**
   不是一次性回答，而是能持续记录、追踪、总结、提醒、归档、复盘、协作。

3. **本地优先**
   核心数据、记忆、文档、画像、时间线、分身人格都以本地文件为事实来源。

4. **多层记忆清晰**
   短期上下文、长期画像、结构化指标、原始病历、周报月报、主动提醒彼此联通但不混乱。

5. **可人格化**
   同一套健康底层可以衍生出：
   - 个人健康管理分身
   - 慢病管理分身
   - 肿瘤患者陪伴分身
   - 家庭照护分身
   - 健康研究与文献分身

6. **强边界与高可信**
   永远不装医生，永远区分记录、解释、建议、风险和必须线下就医的情况。

7. **形成护城河**
   不只是 skill 多，而是：
   - 底层记忆模型强
   - 工作区模板强
   - 多 agent 编排强
   - 健康数据标准强
   - 主动运营能力强

8. **服务普惠化**
   让普通人也能获得过去只有国家领导人、顶级企业家或高净值人群才能长期享有的那种“背后有健康管理团队持续服务”的体验。

## 1. 完美版 VitaClaw 的定义

当以下条件同时满足时，才算接近“完美”：

- 新用户执行一次初始化后，得到完整的 `health` agent workspace
- 默认自带 `AGENTS.md / SOUL.md / IDENTITY.md / USER.md / MEMORY.md / HEARTBEAT.md / BOOTSTRAP.md`
- 默认自带 `memory/health/` 全套目录、模板、样例和说明
- 至少 10 个核心健康技能可真实运行、可写回 memory、可相互读取
- 至少 3 个场景型健康分身可直接上手
- 周报、月报、复查提醒、异常预警能自动形成闭环
- 患者档案、健康时间线、可穿戴数据、日常指标能统一归档和检索
- 多 agent 架构下，个人健康、家人照护、研究模式彼此隔离
- 文档、技能数量、依赖、许可证、测试结果全部一致
- 用户第一次使用时能明显感受到“这不是 skill 仓库，而是一个真的健康助理”
- 用户逐渐感受到背后像是有一个虚拟健康团队在持续协作，而不是单点问答机器人

## 1.1 愿景升级：从健康助手到“虚拟健康管理团队”

真正高级的健康服务，核心从来不是“某个名医随时在线”，而是背后那套持续运转的系统：

- 长期档案有人维护
- 指标波动有人盯
- 异常变化有人预警
- 检查结果有人归档和对比
- 就医前有人准备 briefing
- 就医后有人追踪执行
- 不同问题有人分工处理

VitaClaw 应该把这套服务流程软件化、模板化、普惠化。

也就是说，我们要做的不是一个“会回答健康问题的 AI”，而是一个面向普通人的：

- 虚拟家庭医生办公室
- 虚拟健康管理中心
- 虚拟长期照护协调团队
- 虚拟医疗秘书与研究助理系统

### 1.1.0 产品级一句话原则

> 目标是做到：**比你自己更持续地了解你的健康。**

这是一个比“比你自己还了解你的健康”更稳、更可信的产品表达。

它的真实含义不是：

- AI 比你更有主权
- AI 比医生更权威
- AI 替你做最终医疗决定

它的真实含义是：

- 比你记得更全
- 比你记录得更连续
- 比你整理得更系统
- 比你更早看到趋势变化
- 比你更少遗漏复查、用药、风险和历史细节
- 在你需要就医、复盘、决策时，能比你自己更快调出完整健康上下文

换句话说，VitaClaw 的理想状态不是“替代你”，而是：

- 成为你健康信息的第二大脑
- 成为你健康管理的长期操作系统
- 成为你背后的虚拟健康管理团队中枢

### 1.1.1 这个“虚拟团队”应当模拟哪些角色

- [ ] 健康总管
  统筹长期画像、计划、提醒、优先级

- [ ] 病历秘书
  负责归档、命名、索引、时间线整理、就诊材料准备

- [ ] 指标分析师
  负责血压、血糖、睡眠、体重、肾功能、血脂等纵向趋势判断

- [ ] 风险雷达
  负责危急值、异常波动、断档、依从性下降、复查逾期预警

- [ ] 生活方式教练
  负责饮食、运动、睡眠、咖啡因、减重、压力管理

- [ ] 心理健康守门员
  负责危机识别、情绪趋势、睡眠情绪联动、求助升级

- [ ] 医疗研究助理
  负责指南、文献、试验、药物相互作用、第二阅读材料准备

- [ ] 就医参谋
  负责门诊前摘要、问题清单、复查项目清单、就医后待办追踪

### 1.1.2 为什么这件事是可行的

因为大多数“领导人级健康服务”的真正壁垒不在于神秘知识，而在于：

- 连续记录
- 结构化归档
- 长期跟踪
- 风险分层
- 多角色协同
- 高执行力

这些恰恰是本地文件、OpenClaw workspace、多 agent、memory、heartbeats、automation 最适合做的事情。

### 1.1.3 必须明确的边界

这不是“国家领导人同款医疗资源复制器”。

它真正普惠化的是：

- 服务流程
- 信息组织
- 连续追踪
- 风险提醒
- 决策准备
- 健康管理执行力

不是：

- 特权挂号
- 真实名医替身
- 处方越权
- 绕过医疗体系

所以 VitaClaw 要做的是把“高水平健康管理流程”民主化，而不是把“医疗权威”假装自动化。

## 2. 当前问题总览

### P0 级现状问题

- README 标称 204 个技能，但仓库实际为 207 个，文档与代码漂移
- 多个 Python 技能依赖 `skills/_shared`，但当前仓库缺失该目录
- 不少技能只有 `SKILL.md`，缺少真实可运行脚本、测试或依赖说明
- 仓库尚未交付 OpenClaw 官方 workspace 文件模板的健康版定制
- `health-memory` 有协议，但没有成为项目级统一事实层
- “场景编排”大多还停留在工作流说明，而不是真正可安装即用的产品模板

### 结构性问题

- 过于强调 skill 数量，弱化了“操作系统式”的健康底座
- 缺少统一 schema、统一 shared library、统一 smoke test
- 缺少从 OpenClaw 官方 bootstrap 机制直接衔接的 onboarding 体验
- 缺少 per-agent workspace 打包和分身人格策略
- 缺少明确的商业级/产品级“健康分身”定义与交付形态

目标：不只是一个健康 agent，而是一组可分工协作、可隔离的健康 agents

### 8.1 定义标准 agent 套件

- [x] `health-main`
  个人健康总管，面向本人日常使用

- [x] `health-research`
  只做文献、指南、试验检索，不读个人长期记忆

- [x] `health-family`
  面向家人照护或家庭群，权限更低，输出更保守

- [ ] `health-oncology`
  肿瘤、病历、时间线、治疗相关数据整合

- [ ] `health-mental`
  情绪、睡眠、危机识别优先

### 8.1.1 进一步升级为“虚拟健康管理团队”

在标准 agent 套件之上，进一步定义可协作的虚拟团队角色：

- [ ] `health-chief-of-staff`
  负责总协调、优先级、健康计划、跨 agent 调度

- [ ] `health-records`
  负责文档归档、病历索引、时间线、门诊资料包

- [ ] `health-metrics`
  负责日常指标、趋势、周报、月报、断点识别

- [ ] `health-lifestyle`
  负责饮食、运动、睡眠、体重、行为改变支持

- [ ] `health-safety`
  负责危急值、精神健康危机、药物风险、复查过期预警

- [ ] `health-research`
  负责外部指南、文献、药物与试验检索

目标不是让用户直接面对 6 个机器人，而是让 `health-main` 或 `health-chief-of-staff` 在背后像一个团队经理一样调用它们。

### 8.1.2 虚拟团队的交付形态

- [ ] 单聊天入口
  用户只和一个主分身对话

- [ ] 后台多角色协同
  由主分身按需调用其它专长 agent

- [ ] 自动形成团队产物
  例如：
  - 周报
  - 月报
  - 复查提醒
  - 门诊前 briefing
  - 检查结果差异对比
  - 风险清单
  - 行动计划

- [ ] 面向普通人极低门槛
  用户不需要理解 agent、memory、workspace，只需要感受到“背后有人在持续盯着”

### 8.2 每个 agent 都有独立 workspace

- [x] 每个 agent 独立 `SOUL.md`
- [x] 每个 agent 独立 `MEMORY.md`
- [x] 每个 agent 独立 `HEARTBEAT.md`
- [ ] 每个 agent 独立 tool policy
- [ ] 每个 agent 独立 sandbox 策略

### 8.3 默认安全策略

- [ ] `health-main`：主 agent，可读写健康 workspace
- [ ] `health-family`：默认 sandbox + read-only / limited write
- [ ] `health-research`：尽量无个人健康数据，只做外部研究
- [ ] 群聊模式禁载 `MEMORY.md`
- [ ] 公共场景不自动暴露长期健康记忆

### 8.4 分身人格矩阵

- [ ] 总管型：清晰、稳重、持续跟进
- [ ] 研究型：证据优先、引用清晰、少情绪表达
- [ ] 陪伴型：温和、低压、鼓励坚持
- [ ] 照护型：高可靠、高可执行、强调照护流程

验收标准：

- VitaClaw 不再是一个 skill 大仓库，而是有标准化 agent lineup 的系统

## 9. Phase 5：主动运营与自动化

目标：让健康分身从“被动问答”升级为“主动管理”

### 9.0 主动服务原则

VitaClaw 不能把主动能力理解成“偶尔发一个提醒”，而应当把它理解成持续服务能力。

核心原则：

- [ ] 不依赖用户每次手动触发
- [x] 有明确触发条件，而不是随机打扰
- [x] 有服务优先级，而不是所有提醒同等重要
- [x] 有低打扰机制，而不是频繁轰炸
- [x] 有闭环，而不是只提醒不跟进
- [x] 有个体化，而不是每个人都收到同一套模板提醒

主动服务触发源至少分为四类：

- [x] 时间触发
  例如每天、每周、每月、复查前 3 天

- [ ] 事件触发
  例如新体检报告导入、药物调整、门诊结束、Apple Health 数据更新

- [x] 状态触发
  例如连续 3 天高血压、睡眠明显恶化、体重反弹、依从性下降

- [x] 缺失触发
  例如该记录的数据没有记录、周报没有生成、某项检查长期未更新

### 9.0.1 主动服务不等于骚扰

要明确“该提醒”和“不该提醒”：

- [x] 紧急风险优先强提醒
- [x] 重要但不紧急事项采用温和提醒
- [x] 同类提醒避免短时间重复发送
- [x] 用户已确认/已完成的事项自动静默
- [x] 夜间和用户忙碌时段默认降低打扰强度
- [x] 允许用户配置提醒频率、渠道、语气和安静时段

### 9.0.2 主动服务的真正交付物

主动服务不应只输出一句“请注意”，而应尽量输出：

- [x] 为什么提醒你
- [x] 依据是什么
- [x] 风险级别是什么
- [x] 现在最该做什么
- [ ] 如果不处理，下一步会发生什么
- [x] 我会在什么时候再次跟进

### 9.1 HEARTBEAT 驱动的后台健康维护

- [x] 每日检查是否有缺失记录
- [x] 每周生成健康周报
- [x] 每月生成健康月报
- [x] 定期回顾 `daily/` 并沉淀到 `MEMORY.md`
- [x] 定期检查复查日程和药物库存

### 9.1.1 HEARTBEAT 的职责边界

HEARTBEAT 更适合做“轻量后台巡检”：

- [x] 数据是否断档
- [x] 是否有待整理病历
- [x] 是否到周报/月报节点
- [x] 是否有即将到期的复查或用药事项
- [x] 是否需要把最近变化蒸馏进 `MEMORY.md`

不适合只靠 HEARTBEAT 的任务：

- [ ] 精准到分钟的提醒
- [ ] 明确日期的复查预约通知
- [ ] 必须独立会话执行的后台报告任务

这些应交给 automation / cron 层。

### 9.2 自动化能力

- [x] 复查提醒
- [x] 药物续配提醒
- [x] 年度体检提醒
- [x] 异常指标复核提醒
- [x] 周/月汇总自动产出
- [x] 病历新文档归档提醒

### 9.2.1 主动提醒服务矩阵

至少要实现以下 12 类主动服务：

- [x] 每日记录提醒
  今天还没记录血压/血糖/体重/睡眠时提醒

- [x] 缺失数据催办
  某个关键指标连续缺失 N 天时提醒补录

- [x] 复查到期提醒
  例如肾功能、HbA1c、血脂、肿瘤标志物、影像复查

- [x] 药物续配提醒
  根据库存或处方周期提醒续药

- [ ] 周报推送
  自动发送本周趋势、亮点、问题和下周计划

- [ ] 月报推送
  自动汇总月度趋势、依从性、体重变化、风险变化

- [x] 异常波动提醒
  当指标较基线明显恶化时主动告知

- [ ] 行为计划提醒
  提醒今天的运动、睡眠、饮食、饮水、测量计划

- [x] 门诊前准备提醒
  自动准备门诊摘要与问题清单

- [x] 门诊后执行提醒
  医嘱、复查、调整计划、待办追踪

- [x] 新文档归档提醒
  用户上传了新报告但尚未分类整理时提醒归档

- [ ] 高风险升级提醒
  触发危急值、精神健康危机、严重依从性问题时升级通知

### 9.2.2 提醒必须可个性化

- [x] 允许用户选择提醒频率
- [x] 允许用户选择提醒渠道
- [x] 允许用户选择提醒语气
- [ ] 允许用户定义哪些问题要“盯紧一点”
- [ ] 允许用户定义哪些问题“只在高风险时提醒”
- [x] 允许用户设置安静时段与工作时段

### 9.3 主动识别健康管理断点

- [x] 连续多天无数据
- [ ] 长期计划未更新
- [x] 最近一次复查已过期
- [ ] 指标改善停滞
- [ ] 睡眠/情绪恶化但未触发正式求助

### 9.3.1 从“识别断点”升级到“推动闭环”

不仅要发现问题，还要做后续动作：

- [x] 发现断点后生成一条明确行动建议
- [x] 若用户未响应，在合理时间后再次跟进
- [x] 若问题持续存在，升级提醒级别
- [x] 若用户已处理，自动归档并静默
- [ ] 将反复出现的断点写入 `MEMORY.md` 的执行障碍画像

### 9.4 Apple Health / Wearable / 体检文档联动

- [x] 打通 `apple-health-digest`
- [x] 打通 `medical-record-organizer`
- [x] 打通 `health-timeline`
- [x] 将可穿戴、病历、指标汇总到统一时间轴

### 9.5 领导人级流程的普惠化复制

把高端健康管理服务拆成标准动作，并让系统自动完成：

- [x] 每日监测
  是否有新数据、是否有缺失、是否有异常

- [x] 每周复盘
  指标、行为、依从性、睡眠、情绪、风险变化

- [x] 每月健康运营
  计划是否执行、哪些问题反复出现、是否需要门诊或复查

- [x] 门诊前 briefing
  自动生成：
  - 最近异常变化
  - 要问医生的问题
  - 当前药物与补充剂
  - 最近检查摘要
  - 建议复查项

- [x] 门诊后 follow-up
  自动记录医生建议、后续待办、复查时间点、需要跟踪的指标

- [ ] 长周期风险管理
  自动维护：
  - 体重控制进度
  - 慢病稳定性
  - 睡眠债
  - 情绪恶化趋势
  - 检查逾期项目

验收标准：

- 用户不发消息时，分身仍能通过 heartbeat / automation 做低打扰、真有用的维护
- 用户会感觉自己背后长期站着一个有分工、有记忆、有执行力的健康管理团队
- 即使没有高端私人医疗资源，也能得到接近其“流程质量”和“管理密度”的服务体验

## 17. 结语

VitaClaw 下一阶段最重要的转变，不是“再加更多 skill”，而是：

从：

- 一个健康技能大仓库

升级为：

- 一个可直接拥有健康分身的 OpenClaw 产品模板

再升级为：

- 一个围绕健康记忆、健康画像、长期追踪、主动提醒和多 agent 协作构成的健康操作系统

最终再升级为：

- 一个让普通人也能拥有“虚拟健康管理团队”的普惠型个人健康基础设施
