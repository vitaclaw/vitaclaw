# VitaClaw Digital Anatomy 2 逐字稿

## 逐字稿

### 第 1 页：The Health Cognition Layer for Personal AI Agents

今天我想介绍的，不是一个新的健康聊天机器人，而是一层面向个人 AI Agent 的健康认知层。

我们把这套体系拆成两个名字来看：`VitaClaw` 是开源的技能库，`Vita Protocol` 是底下那套本地文件化的健康上下文协议。

它想解决的问题，不是“AI 能不能回答一个健康问题”，而是“AI 能不能在长期使用中持续地记住、整理、追踪、调取，并且审计一个人的健康上下文”。

所以我们从一开始就没有把它设计成一个只在单轮对话里工作的问答系统，而是设计成一个 file-based、LLM-native、local-first 的健康工作区。

在这个工作区里，AI 不是每次从零开始理解用户；它可以回到本地 `memory/health/`，读取健康画像、最近 daily、相关指标文件和必要的临床归档，再做连续判断。

一句话说，这一页的核心信息是：VitaClaw 不是 another health chatbot，它是一层给个人 AI Agent 使用的健康认知层。

### 第 2 页：The Core Architectural Mismatch

为什么我们要重新设计底层？因为我们认为这里存在一个很典型的架构错配。

在过去的软件时代，默认前提是：你有一个 EHR 或 FHIR 数据库，前端应用可以用几乎无限次的 REST API 查询，按字段、按资源、按条件不断取数。

这套范式对传统软件当然成立，而且今天也仍然成立。我们并没有否认 FHIR、HL7 或机构数据库的价值。恰恰相反，在当前仓库里，FHIR 仍然被视为机构互操作的重要基础。

但问题在于，LLM Agent 的工作方式不是这样。Agent 真正受限的不是“有没有 API”，而是“当前上下文里到底能放下多少有用信息”。

如果我们把面向机构系统设计的原生对象、长 JSON、碎片化接口调用结果，直接当作 Agent 的认知输入，成本会很高，语义密度也不稳定。

所以对 Agent 来说，只给一个通信 API 不够。它需要的是一个可被它直接读取、按需展开、长期沉淀、可审阅、可写回的本地认知系统。

### 第 3 页：The Paradigm Shift: From Software to Agent

因此，我们真正做的不是简单替代，而是一次范式加层。

对传统软件来说，最自然的数据消费者是 Web 或 Mobile App；对未来的个人 AI 来说，最自然的数据消费者正在变成长期运行的 LLM Agent。

这意味着表达方式也要变化。Agent 更适合直接消费结构清楚、语义密度高、可局部读取的本地文本，而不是每次都重新从冗长接口对象里拼装含义。

VitaClaw 在当前仓库里已经很明确地把 `Read / Grep / Glob / Write / Edit` 这类文件操作放在第一层，把 memory contract 放在多技能共享的中心位置。

但我这里也会更谨慎一点说明：当前实现不是“纯 Markdown 取代一切”。

更准确地说，它是 `Markdown-first` 的本地认知平面，同时保留 JSONL 结构化记录层，也保留 XML 和 JSON 的桥接入口，比如 Apple Health 导入和同步桥。

我们不是从软件世界跳到一个完全抛弃结构化数据的世界，而是在机构系统之上，为 Agent 增加了一层更适合推理和长期记忆的本地语义层。

### 第 4 页：The Vita Protocol Architecture

接下来这页是整个 Vita Protocol 的整体架构。

第一层是 `Schema` 层。当前仓库里，最核心的四类本地结构非常清楚：长期健康画像、每日记录、单项纵向追踪，以及患者档案或临床归档。

第二层是 `Protocol` 层。也就是固定读取顺序、增量读写规则、写回边界、daily 到 MEMORY 的蒸馏链，以及不同角色之间的路由与写回约束。

第三层是 `Trust` 层。这里如果完全按仓库现状来讲，我会更落地一些：当前已经比较明确落地的，不是一个完备的 T1 到 T5 引擎，而是来源、证据、隐私分层、写回审计、调度审计和“不允许直接把推断写成长期事实”的一整组防护约束。

第四层是 `Sovereignty` 层。它的核心不是一句口号，而是一个工程选择：关键健康上下文默认落在用户本地文件系统里，能被人类审阅，也能被 agent 读取，还能留下清晰的工件和审计痕迹。

所以这一页你可以这样收束：

Vita Protocol 不是一个单点格式，而是一组围绕本地 schema、增量协议、可信边界和文件主权组织起来的健康认知架构。

### 第 5 页：The Health Memory System: Dual-Axis Storage

这一页是整个设计里我认为最重要、也最容易讲清楚的一页。

我们问自己一个非常具体的问题：如果用户只想知道“我的血压最近有没有改善”，为什么要逼一个大模型把过去九十天所有 daily 文件全读一遍？

VitaClaw 给出的答案是双轴记忆。

X 轴是时间轴，也就是 `daily/`。它把同一天来自多个 skills 的数据聚合到一个文件里，适合回答“昨天发生了什么”“今天做了什么”“这次体感和执行怎么样”。

Y 轴是指标轴，也就是 `items/`。它把某一个具体指标的最近状态、历史表、阈值和滚动摘要集中维护，适合回答“我的血糖有没有改善”“最近体重趋势怎样”“肾功能有没有持续波动”。

这两个轴不是二选一，而是互补的。一个负责横向聚合同一天发生的事情，一个负责纵向压缩同一指标跨时间的变化。

更关键的是，它们通过统一格式契约相连：每个 skill 在 daily 里写自己的标准 section，每个 item 文件至少有 `Recent Status` 和 `History`，这样不同 skills 之间可以零代码集成。

所以这页真正想表达的是：我们不是单纯把文件分目录，而是在设计一种让 Agent 可以低成本定位“时间上下文”和“指标上下文”的双轴记忆系统。

### 第 6 页：The Incremental Read Protocol

有了双轴存储以后，还需要一套真正可执行的读取协议。

VitaClaw 在这件事上采取的思路非常朴素，但非常适合 Agent：先加载最小必要上下文，再按问题逐层展开。

第 0 层通常是健康画像。第 1 层是最近的 daily。只有当用户问到具体指标时，才继续读取相关 item 文件。只有当用户追问具体历史日期、病历、时间线、原始报告时，才继续深入到历史文件和患者档案。

在当前仓库里，这套逻辑已经被明确写成固定读取顺序和 `Read Protocol`，而且工具本身就是 `Read / Grep / Glob` 这种对 Agent 极其友好的轻量能力。

我建议你在这一页把重点放在“分级加载”和“避免无意义装载”上，而不是死守某一个具体节省百分比。

因为从现有仓库来看，增量读取这件事是明确存在的；但如果我们要说某个严格的 benchmark 数字，最好单独有实验出处。

所以更稳妥、也更真实的讲法是：

这套协议的价值在于，它让 Agent 默认读取的是最小必要上下文，而不是一次性吞下整个健康记忆仓库。

### 第 7 页：Trust, Provenance, and File Sovereignty

健康 AI 不是只要“会分析”就够了，真正困难的是：它怎么知道什么是原始证据，什么是用户自报，什么是设备数据，什么只是模型推断。

这一页表达的是我们对“信任”和“主权”的两个基本立场。

先说主权。当前仓库已经非常明确地站在 local-first 这一边：健康记忆、daily、items、brief、audit、timeline 这些关键工件默认都落在本地文件里，而不是云端黑盒。

再说信任。按仓库现状，我会更谨慎地讲：我们已经把 `Sources`、`Evidence`、`write-audit`、`dispatch-log`、`privacy_tier`、`execution_mode` 这些可追溯字段落进了 protocol 和 runtime 里。

这意味着一个好的输出，不只是给一个结论，而是要同时告诉你：我看了哪些文件，我依据了哪些阈值或规则，我写回了哪些本地文件，我把什么任务分派给了后台角色。

至于图里这套 T1 到 T5 的信任分层，我会把它讲成 VitaClaw 的目标表达：实验室结果、验证设备、结构化自报、非结构化感受和 agent 派生结论，在临床意义上绝不应该被等同对待。

换句话说，这一页真正想建立的不是“AI 很聪明”，而是“AI 必须知道自己凭什么说这句话，而且必须让人追得回去”。

### 第 8 页：The VitaClaw Library

在协议底座之上，VitaClaw 不是只停留在抽象架构，而是扩展成了一个实际可用的技能生态。

这里我建议你把 slide 上的 `204` 改成更稳妥的说法：`220 多个技能模块`。如果你希望报当前仓库数字，那按我现在看到的仓库快照，是 `225`。

这些 skills 的价值不只是在数量上，更在于覆盖面和互操作性上。

它一端连接日常健康管理，比如血压、血糖、睡眠、体重、补剂、心理支持和危机检测；另一端也连接研究和专科能力，比如 Apple Health、PubMed、药物数据库、肿瘤标志物、时间线、基因组与变异解释。

但真正让这个生态成立的，不是“有很多 skill”，而是“这些 skill 共享同一套本地记忆契约”。

也就是说，一个 skill 写下来的 daily section 和 item history，另一个 skill 可以直接继续读、继续分析、继续写回，而不需要为每一对技能写一层转换器。

所以这页我建议你用一句很清晰的话总结：

VitaClaw 的重点不是 skill marketplace，而是一个共享 memory contract 驱动的可组合健康能力层。

### 第 9 页：Scenario Orchestration: The Diabetes Control Hub

这一页最适合用来说明：VitaClaw 不是把很多技能堆在那里，而是把它们编排成一个能闭环的场景。

以 Diabetes Control Hub 为例，起点不是复杂建模，而是一个非常真实的触发器：用户记录了一次晨起血糖、一顿饭，或者一次运动。

接着系统先过安全门。只要出现明确的低血糖阈值，比如低于 `3.9 mmol/L`，就优先进入必须就医或立即处理的路径，而不是继续做漂亮但危险的后续推演。

安全通过以后，场景才会继续串联多个子能力：condition monitor 负责血糖趋势、eHbA1c 和 TIR；nutrition 看饮食结构和升糖线索；fitness 看运动和变化关联；必要时再把体重、肾功能、复查计划和行为计划一起纳入。

最后不是停在“分析完了”，而是执行写回协议：把结果落到 daily，把纵向摘要写进 `blood-sugar.md`、`weight.md`，把 follow-up 推到 task board，把写回动作记录到审计文件里。

这里我建议你刻意强调两个词：`Safety Gate` 和 `Write-back`。

因为这两个词最能说明 VitaClaw 的差异：它不是只会分析，它还会先做安全分流，再把结果沉淀进长期可用的本地健康上下文。

### 第 10 页：The AI-Era Health Stack

最后这张图是整个叙事的收束。

它表达的不是“AI 时代不需要医院系统了”，恰恰相反，它表达的是分层协同。

最底层，机构系统仍然是地基。FHIR、HL7、医院记录、法定病历，这些依然是系统间互操作和正式记录的基础。

中间层，是 ingest 和 export 的桥。它的职责是把外部系统里的结构化资源，桥接、提取、压缩成更适合个人 agent 使用的本地健康上下文。

最上层，才是 personal agent cognition。也就是用户自己的 AI agent，基于本地文件和长期记忆，持续地记录、追踪、提醒、复盘和跟进。

所以如果我要用一句话为整套演讲收尾，我会这样讲：

VitaClaw 想做的，不是替代医院系统，也不是做一个一次性健康问答机器人；它想做的是 AI 时代个人健康栈里的认知层，让用户第一次真正拥有一个能长期工作的本地健康 agent。

## 收尾版一句话

如果现场只允许我留下一句总结，那我会说：

`VitaClaw is a local-first health cognition layer for personal AI agents: not just answering health questions, but persistently organizing, retrieving, and auditing health context over time.`
